# Per-provider CLI login reference

Verified by probing each CLI (help + source + shadowed-`open` tests), and **most rows were completed end-to-end with a real human login this round** (Sanity, Netlify, Vercel, Cloudflare, Neon, Render, WorkOS, Railway, Daytona, Fly, Convex, Turso, Auth0, Stripe, Stack Auth, Modal, Supabase, + E2B already-authed; Clerk confirmed API-key-only). Method is in `SKILL.md`; this is the lookup table.

**Legend**
- ЁЯЯв default `login` auto-opens the popup **and** stores the token itself тАФ the happy path.
- ЁЯФ╡ **device-code** тАФ shows a `user_code`/URL to confirm in the browser (no localhost callback); surface the code to the human before they approve.
- тМия╕П **needs a real TTY** тАФ bare/backgrounded login hits an interactive prompt and *fatals*. Drive it in a real-terminal foreground, with `npx noninteractive` (`<pkg>` for npm CLIs, `start <binary>` for binaries тАФ then `send` the keystroke), or `expect`.
- ЁЯФЧ **prints the URL only** (no auto-open) тАФ you must surface the URL to the human yourself.
- ЁЯФС **API-key-only** тАФ no browser/OAuth flow exists; authenticate by setting/pasting an API key (e.g. Clerk: `clerk init` / `clerk config set clerk.key`). Here the `token` env var **is** the login path, not a fallback.
- `token` = the env var that skips the popup entirely (CI / headless / no-popup fallback).

## The тМия╕П TTY-gated logins (where a bare/backgrounded run fatals)

These show an interactive prompt *before* the popup, so a non-TTY run errors out. Run them in a real-terminal foreground, via `npx noninteractive` (`<pkg>` or `start <binary>`) or `expect`, sending the keystroke:

- **Railway** тАФ needs a TTY (non-TTY: `Cannot login in non-interactive mode`); v5.x auto-opens under a PTY with no keystroke (old v4.x prompts `Open the browser? (Y/n)`). v5 auto-detects SSH/CI/no-DISPLAY and switches to device-code itself тАФ pass `railway login --browserless` (pairing code; also needs a TTY) explicitly only when the box has no browser.
- **Daytona** тАФ TUI `Select Authentication Method` тЖТ `Login with Browser` (press Enter).
- **Render** *(v1.x only)* тАФ default `--output interactive` runs a post-login workspace picker that fatals non-TTY (`panic: could not open /dev/tty`); the token is saved BEFORE the panic, so a backgrounded `render login` still succeeds тАФ pass `-o text` to skip it, then confirm the printed device code. **v2.x auto-switches to text on non-TTY**, so no panic (the `-o text` flag stays harmless).
- **Auth0** тАФ survey `How would you like to authenticate? As a user / As a machine`, then `Press Enter to open the browser`.
- **Stripe** *(NOT actually gated in v1.40.9+ тАФ listed here to correct the myth)* тАФ no Enter gate: non-TTY auto-emits JSON and exits 0, so `expect`/`noninteractive` does NOT apply. Run `stripe login --non-interactive`, **surface the `browser_url` for the human + the `verification_code`**, then `stripe login --complete '<browser_url>'` (~130s poll; **on timeout re-run `--complete` with the SAME url тАФ do NOT restart `--non-interactive`**, a fresh url orphans the approval). An exit 0 after the JSON is step 1, not success тАФ verify with `stripe get /v1/account`. (Full detail in the Stripe row above.)
- **Supabase** *(paste-back тАФ use `noninteractive`, not a one-shot `expect`)* тАФ needs a TTY (non-TTY fatals `Cannot use automatic login flow inside non-TTY environments`). `Press Enter to open browserтАж` тЖТ send Enter тЖТ the browser shows a verification code to **enter back into the CLI** ("Enter this verification code in Supabase CLI") тЖТ relay it (text, not a screenshot) via `noninteractive send <sess> <code>` **then send Enter** to submit (~60s TTL; if `unauthorized`, the account hit its token cap тЖТ delete old tokens). (`--no-browser` instead prompts `Enter your access token` for a PAT; skip with `--token`/`SUPABASE_ACCESS_TOKEN`.)
- **Convex** тАФ TWO gates: a `Device name:` prompt (skip with `--device-name <name>`) **and** a following `Open the browser? (Y/n)` prompt that also needs a TTY, so `--device-name` alone still fatals non-interactive. Drive the `Open the browser?` answer with `expect`/`noninteractive` тАФ the generic `expect` recipe's `open the browser` match already handles it.
- **Sanity** тАФ only the provider picker is interactive; `--provider google|github` skips it and the rest is hands-free.
- **Stack Auth** *(not a prompt, but тМия╕П-gated all the same)* тАФ it **buffers the login URL on non-TTY stdout** (even `stdbuf -oL` won't flush it), so run under a PTY (`expect`/`noninteractive`) to surface the URL.

## Common popup-suppressors to NOT set

`CI=1` (Vercel, WorkOS, Neon тЖТ URL prints but no popup) ┬╖ `BROWSER=none` / bogus `$BROWSER` ┬╖ `--no-open` / `--no-browser` / `--headless` / `--browserless` / `--non-interactive` ┬╖ a token env var already set (most CLIs skip login when their `*_TOKEN`/`*_API_KEY` is present тАФ unset it to force the popup). Stored creds in the CLI's *config file* (not just env) can also short-circuit login or block verify (e.g. Daytona's `config.json` `api.key`) тАФ clear those too if a login mysteriously no-ops. Daytona's config is `~/Library/Application Support/daytona/config.json` (macOS) / `~/.config/daytona/config.json`; token = the profile's `api.token` (NOT the keychain), and `daytona logout` clears the token but leaves a stale `api.url` тАФ a leftover `http://localhost:3000` makes login *succeed* yet `daytona organization list` fail `connection refused` (delete the config to reset).

**Exception тАФ Turso `--headless`:** the blanket `--headless` ban above is the rule for auto-open CLIs, but Turso *documents* `turso auth login --headless` as its WSL/remote/SSH fallback тАФ it prints the auth URL to open manually instead of trying (and hanging on) a browser auto-open. In exactly the headless/agent contexts this skill targets, prefer it over fighting the auto-open.

**Exception тАФ Railway `--browserless`:** the Railway row offers `railway login --browserless` (pairing code) as the no-browser fallback, so it's exempt from the `--browserless` ban above. **Caveat:** it's been **flaky for agents** (may not open a popup) тАФ last resort only, and re-verify with `railway whoami`.
