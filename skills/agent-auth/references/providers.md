# Per-provider CLI login reference

Verified by probing each CLI (help + source + shadowed-`open` tests), and **most rows were completed end-to-end with a real human login this round** (Sanity, Netlify, Vercel, Cloudflare, Neon, Render, WorkOS, Railway, Daytona, Fly, Convex, Turso, Auth0, Stripe, Stack Auth, Modal, Supabase, + E2B already-authed; Clerk confirmed API-key-only). Method is in `SKILL.md`; this is the lookup table.

**Legend**
- рЯЯҐ default `login` auto-opens the popup **and** stores the token itself вАФ the happy path.
- рЯФµ **device-code** вАФ shows a `user_code`/URL to confirm in the browser (no localhost callback); surface the code to the human before they approve.
- вМ®пЄП **needs a real TTY** вАФ bare/backgrounded login hits an interactive prompt and *fatals*. Drive it in a real-terminal foreground, with `npx noninteractive` (`<pkg>` for npm CLIs, `start <binary>` for binaries вАФ then `send` the keystroke), or `expect`.
- рЯФЧ **prints the URL only** (no auto-open) вАФ you must surface the URL to the human yourself.
- рЯФС **API-key-only** вАФ no browser/OAuth flow exists; authenticate by setting/pasting an API key (e.g. Clerk: `clerk init` / `clerk config set clerk.key`). Here the `token` env var **is** the login path, not a fallback.
- `token` = the env var that skips the popup entirely (CI / headless / no-popup fallback).

The **Light install** cells append `--version` on purpose: it warms the npx/pip cache and prints the version **without side effects**. Running the bare package instead can *do something* вАФ `npx -y vercel` deploys, `npx -y workos@latest` runs the installer вАФ so never strip the trailing flag (see `SKILL.md`).

**`npx` doesn't leave the CLI on `PATH`.** The Login/Verify cells show the bare command (`sanity login`, `wrangler whoami`) for readability, but for an npx-install row on a fresh machine, run them the **same npx way** вАФ `npx -y <pkg>@latest <subcommand>` (e.g. `npx -y @sanity/cli@latest login --provider google`, `npx -y wrangler@latest whoami`) вАФ or `npm i -g <pkg>` once to get a real `PATH` binary. Bare `sanity`/`wrangler` after only an npx call is `command not found`. The `pip`/`brew`/`curl` rows *do* install a real binary, so their bare commands work as written (mind the PATH caveats noted per-row).

## The вМ®пЄП TTY-gated logins (where a bare/backgrounded run fatals)

These show an interactive prompt *before* the popup, so a non-TTY run errors out. Run them in a real-terminal foreground, via `npx noninteractive` (`<pkg>` or `start <binary>`) or `expect`, sending the keystroke:

- **Railway** вАФ needs a TTY (non-TTY: `Cannot login in non-interactive mode`); v5.x auto-opens under a PTY with no keystroke (old v4.x prompts `Open the browser? (Y/n)`). v5 auto-detects SSH/CI/no-DISPLAY and switches to device-code itself вАФ pass `railway login --browserless` (pairing code; also needs a TTY) explicitly only when the box has no browser.
- **Daytona** вАФ TUI `Select Authentication Method` вЖТ `Login with Browser` (press Enter).
- **Render** *(v1.x only)* вАФ default `--output interactive` runs a post-login workspace picker that fatals non-TTY (`panic: could not open /dev/tty`); the token is saved BEFORE the panic, so a backgrounded `render login` still succeeds вАФ pass `-o text` to skip it, then confirm the printed device code. **v2.x auto-switches to text on non-TTY**, so no panic (the `-o text` flag stays harmless).
- **Auth0** вАФ survey `How would you like to authenticate? As a user / As a machine`, then `Press Enter to open the browser`.
- **Stripe** *(NOT Enter-gated вАФ non-TTY auto-emits JSON and exits 0, even under a PTY: the npm shim's stdin isn't seen as a terminal)* вАФ `expect`/`noninteractive` can't drive it with a keystroke (it exits immediately), **but `npx noninteractive @stripe/cli@latest login` auto-opens the `browser_url` confirm page** (nicer than surfacing the URL by hand; expect a junk `вА¶/auth/cliauthвА¶?secret=вА¶` tab too). Surface the `verification_code`, then run the `next_step` `stripe login --complete '<next_step url вАФ the cliauth?secret= one, NOT browser_url>'`, **re-arming until it captures** (the poll spanning the human's approval saves the key; a one-shot times out and strands it). Verify with `stripe get /v1/account`. (Full detail in the Stripe row above.)
- **Supabase** *(paste-back вАФ use `noninteractive`, not a one-shot `expect`)* вАФ needs a TTY (non-TTY fatals `Cannot use automatic login flow inside non-TTY environments`). `Press Enter to open browserвА¶` вЖТ send Enter вЖТ the browser shows a verification code to **enter back into the CLI** ("Enter this verification code in Supabase CLI") вЖТ relay it (text, not a screenshot) via `noninteractive send <sess> <code>` **then send Enter** to submit (~60s TTL; if `unauthorized`, the account hit its token cap вЖТ delete old tokens). (`--no-browser` instead prompts `Enter your access token` for a PAT; skip with `--token`/`SUPABASE_ACCESS_TOKEN`.)
- **Convex** вАФ TWO gates: a `Device name:` prompt (skip with `--device-name <name>`) **and** a following `Open the browser? (Y/n)` prompt that also needs a TTY, so `--device-name` alone still fatals non-interactive. Drive the `Open the browser?` answer with `expect`/`noninteractive` вАФ the generic `expect` recipe's `open the browser` match already handles it.
- **Sanity** вАФ only the provider picker is interactive; `--provider google|github` skips it and the rest is hands-free.
- **Stack Auth** *(not a prompt, but вМ®пЄП-gated all the same)* вАФ it **buffers the login URL on non-TTY stdout** (even `stdbuf -oL` won't flush it), so run under a PTY (`expect`/`noninteractive`) to surface the URL.

## Common popup-suppressors to NOT set

`CI=1` (Vercel, WorkOS, Neon вЖТ URL prints but no popup) ¬Ј `BROWSER=none` / bogus `$BROWSER` ¬Ј `--no-open` / `--no-browser` / `--headless` / `--browserless` / `--non-interactive` ¬Ј a token env var already set (most CLIs skip login when their `*_TOKEN`/`*_API_KEY` is present вАФ unset it to force the popup). Stored creds in the CLI's *config file* (not just env) can also short-circuit login or block verify (e.g. Daytona's `config.json` `api.key`) вАФ clear those too if a login mysteriously no-ops. Daytona's config is `~/Library/Application Support/daytona/config.json` (macOS) / `~/.config/daytona/config.json`; token = the profile's `api.token` (NOT the keychain), and `daytona logout` clears the token but leaves a stale `api.url` вАФ a leftover `http://localhost:3000` makes login *succeed* yet `daytona organization list` fail `connection refused` (delete the config to reset).

**Exception вАФ Turso `--headless`:** the blanket `--headless` ban above is the rule for auto-open CLIs, but Turso *documents* `turso auth login --headless` as its WSL/remote/SSH fallback вАФ it prints the auth URL to open manually instead of trying (and hanging on) a browser auto-open. In exactly the headless/agent contexts this skill targets, prefer it over fighting the auto-open.

**Exception вАФ Railway `--browserless`:** the Railway row offers `railway login --browserless` (pairing code) as the no-browser fallback, so it's exempt from the `--browserless` ban above. **Caveat:** it's been **flaky for agents** (may not open a popup) вАФ last resort only, and re-verify with `railway whoami`.

**Exception вАФ Stripe `--non-interactive`:** the `--non-interactive` ban targets CLIs where the flag *bypasses* the browser login. Stripe doesn't skip auth вАФ `stripe login --non-interactive` forces the JSON payload (`browser_url` + `verification_code` + `next_step`) you relay to the human, then you finish the real login with `stripe login --complete '<url>'`. (On a non-TTY stdin Stripe auto-selects that JSON mode anyway вАФ the flag just makes it explicit; either way Stripe does **not** auto-open, so you surface the URL yourself.) So Stripe is exempt вАФ it's the command the row needs to get the relay payload, not a popup-suppressor.
