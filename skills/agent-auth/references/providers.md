# Per-provider CLI login reference

Verified by probing each CLI (help + source + shadowed-`open` tests). Sanity was completed end-to-end live; the rest are mechanism-verified (no real login performed). Method is in `SKILL.md`; this is the lookup table.

**Legend**
- üü¢ default `login` auto-opens the popup **and** stores the token itself ‚Äî the happy path.
- üîµ **device-code** ‚Äî shows a `user_code`/URL to confirm in the browser (no localhost callback); surface the code to the human before they approve.
- ‚å®Ô∏è **needs a real TTY** ‚Äî bare/backgrounded login hits an interactive prompt and *fatals*. Drive it in a real-terminal foreground, with `npx noninteractive` (`<pkg>` for npm CLIs, `start <binary>` for binaries ‚Äî then `send` the keystroke), or `expect`.
- üîó **prints the URL only** (no auto-open) ‚Äî you must surface the URL to the human yourself.
- üîë **API-key-only** ‚Äî no browser/OAuth flow exists; authenticate by setting/pasting an API key (e.g. Clerk: `clerk init` / `clerk config set clerk.key`). Here the `token` env var **is** the login path, not a fallback.
- `token` = the env var that skips the popup entirely (CI / headless / no-popup fallback).

## The ‚å®Ô∏è TTY-gated logins (where a bare/backgrounded run fatals)

These show an interactive prompt *before* the popup, so a non-TTY run errors out. Run them in a real-terminal foreground, via `npx noninteractive` (`<pkg>` or `start <binary>`) or `expect`, sending the keystroke:

- **Railway** ‚Äî needs a TTY (non-TTY: `Cannot login in non-interactive mode`); v5.x auto-opens under a PTY with no keystroke (old v4.x prompts `Open the browser? (Y/n)`). v5 auto-detects SSH/CI/no-DISPLAY and switches to device-code itself ‚Äî pass `railway login --browserless` (pairing code; also needs a TTY) explicitly only when the box has no browser.
- **Heroku** ‚Äî `Press any key to open up the browser to login or q to exit` (raw-mode `stdin`); piped stdin throws.
- **Daytona** ‚Äî TUI `Select Authentication Method` ‚Üí `Login with Browser` (press Enter).
- **Render** *(v1.x only)* ‚Äî default `--output interactive` runs a post-login workspace picker that fatals non-TTY (`panic: could not open /dev/tty`); the token is saved BEFORE the panic, so a backgrounded `render login` still succeeds ‚Äî pass `-o text` to skip it, then confirm the printed device code. **v2.x auto-switches to text on non-TTY**, so no panic (the `-o text` flag stays harmless).
- **Auth0** ‚Äî survey `How would you like to authenticate? As a user / As a machine`, then `Press Enter to open the browser`.
- **Stripe** *(NOT actually gated in v1.40.9+ ‚Äî listed here to correct the myth)* ‚Äî no Enter gate: non-TTY auto-emits JSON and exits 0, so `expect`/`noninteractive` does NOT apply. Run `stripe login --non-interactive`, surface `verification_code`, then `stripe login --complete '<browser_url>'`. An exit 0 after the JSON is step 1, not success ‚Äî verify with `stripe get /v1/account`.
- **Supabase** *(paste-back ‚Äî use `noninteractive`, not a one-shot `expect`)* ‚Äî needs a TTY (non-TTY fatals `Cannot use automatic login flow inside non-TTY environments`). `Press Enter to open browser‚Ä¶` ‚Üí send Enter ‚Üí the browser shows a verification code to **enter back into the CLI** ("Enter this verification code in Supabase CLI") ‚Üí relay it (text, not a screenshot) via `noninteractive send <sess> <code>` **then send Enter** to submit (~60s TTL; if `unauthorized`, the account hit its token cap ‚Üí delete old tokens). (`--no-browser` instead prompts `Enter your access token` for a PAT; skip with `--token`/`SUPABASE_ACCESS_TOKEN`.)
- **Convex** ‚Äî TWO gates: a `Device name:` prompt (skip with `--device-name <name>`) **and** a following `Open the browser? (Y/n)` prompt that also needs a TTY, so `--device-name` alone still fatals non-interactive. Drive the `Open the browser?` answer with `expect`/`noninteractive` ‚Äî the generic `expect` recipe's `open the browser` match already handles it.
- **Sanity** ‚Äî only the provider picker is interactive; `--provider google|github` skips it and the rest is hands-free.

## Common popup-suppressors to NOT set

`CI=1` (Vercel, WorkOS, Neon ‚Üí URL prints but no popup) ¬∑ `BROWSER=none` / bogus `$BROWSER` ¬∑ `--no-open` / `--no-browser` / `--headless` / `--browserless` / `--non-interactive` ¬∑ a token env var already set (most CLIs skip login when their `*_TOKEN`/`*_API_KEY` is present ‚Äî unset it to force the popup). Stored creds in the CLI's *config file* (not just env) can also short-circuit login or block verify (e.g. Daytona's `config.json` `api.key`) ‚Äî clear those too if a login mysteriously no-ops. Daytona's config is `~/Library/Application Support/daytona/config.json` (macOS) / `~/.config/daytona/config.json`; token = the profile's `api.token` (NOT the keychain), and `daytona logout` clears the token but leaves a stale `api.url` ‚Äî a leftover `http://localhost:3000` makes login *succeed* yet `daytona organization list` fail `connection refused` (delete the config to reset).

**Exception ‚Äî Turso `--headless`:** the blanket `--headless` ban above is the rule for auto-open CLIs, but Turso *documents* `turso auth login --headless` as its WSL/remote/SSH fallback ‚Äî it prints the auth URL to open manually instead of trying (and hanging on) a browser auto-open. In exactly the headless/agent contexts this skill targets, prefer it over fighting the auto-open.
