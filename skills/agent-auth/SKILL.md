# Logging a human into a CLI

Goal: make the OAuth **browser popup open automatically**, let the human click Approve, and let the CLI **store the token itself**. You don't copy tokens; you don't drive the browser.

**This is *automatic browser login* ‚Äî for the case where there's no token yet and a real human is here to sign in.** That's the whole premise. **Assume a human is at the keyboard:** they'll click Approve in the browser, confirm any device code (against what the CLI printed), and can relay a code or error back to you as text ‚Äî so when you're stuck or need a code, *ask them*. You make the popup appear and react to the CLI; they approve. Some CLIs now detect an agent and try to *refuse* the browser, demanding a token instead (Supabase `--agent`, the JSON/"agent mode" trend) ‚Äî **override that** (`--agent no`, etc.) to get the human's browser flow. A `*_TOKEN`/`--token` env is only the last-resort fallback when there's genuinely no human/browser; never the preferred route here ‚Äî **except üîë API-key-only CLIs (Clerk) that have no browser login at all, where the key *is* the auth (see the per-provider table for which)**.

## Fast-path (do this every time)

1. **DON'T**: scrape token *values* to copy-paste (`cat ~/.config/.../config.json`) ‚Äî none exist until login. (Checking whether `CI`/`BROWSER`/the provider's `*_TOKEN` are set is fine ‚Äî those suppress the popup; if one is actually set, clear it **off the login command only** with `env -u CI -u BROWSER -u <PROVIDER_TOKEN> <cli> login`, which truly unsets them. Include the provider's own token var (`<PROVIDER_TOKEN>` = the last column of the providers table ‚Äî `VERCEL_TOKEN`, `NETLIFY_AUTH_TOKEN`, `SANITY_AUTH_TOKEN`, ‚Ä¶): a set `*_TOKEN` is the *strongest* suppressor ‚Äî the CLI short-circuits straight to token auth and skips the human browser flow entirely ‚Äî so it must come off alongside `CI`/`BROWSER`. **Apply the same `env -u <PROVIDER_TOKEN>` to the *verify* step too** (`env -u <PROVIDER_TOKEN> <cli> whoami`): `env -u` is scoped to one command, so a session-wide token still gets read by a later `whoami`/`status`, which would then report the *env token's* identity and pass even if the browser login stored nothing ‚Äî a false "logged in." Keep the token cleared across the whole login-and-verify sequence (or `unset` it in the shell for the duration). Don't reach for `CI= BROWSER= <cli> login`: an *empty* `BROWSER=` is itself a popup-suppressor on some CLIs, so you'd re-introduce the very thing you're clearing ‚Äî and only bother clearing at all if it's genuinely set, since needlessly emptying `BROWSER` can suppress an otherwise-fine auto-open.) DON'T drive the browser (xdotool, screenshot the X root, CDP, browser-agent logs). DON'T heavy-install an SDK (`npm i <pkg>`, 1000+ pkgs / ~2 min) just to authenticate. DON'T reach for a provider's MCP auth tool (e.g. `mcp__sanity__authenticate`) ‚Äî drive the CLI's own login.
2. **Get the CLI light & current**: `npx -y <cli-package>@latest --version` (always pair it with `--version`/a subcommand ‚Äî **never run the bare package**: some default binaries *do work*, e.g. bare `vercel` deploys, bare `workos` scaffolds). This also dodges a *stale* global/brew binary already on PATH, which can have a broken login (an old Homebrew Railway **v4.x** fails its callback). **Binary-only CLIs** (flyctl, render, daytona, auth0 ‚Äî no npm package) skip the `npx` step entirely: install the documented `brew`/`curl` binary instead, but still `--version`-check *that* binary and watch for a stale older copy (brew `render` ships v1.x while v2.x exists). Sanity-check `<cli> --version 2>&1 | grep -v 'npm warn'` ‚Äî it MUST print a real version. Blank/error = a stale binary shadowing it (`brew upgrade`/full path) **or** the `npx` wrapper never fetched its native binary (`npx -y @railway/cli` does this ‚Üí `npm i -g <cli> --foreground-scripts`, or brew). A blank `whoami`/`logout` means the same ‚Äî NOT logged-out. **Never `sudo`** to install or fix a CLI ‚Äî on shared Macs `brew` often fails (`/opt/homebrew/Cellar is not writable`) and sudo isn't available; fall back to a user-level install (the CLI's GitHub-release Darwin tarball run from the scratchpad, or a vendor `curl ‚Ä¶ | bash` that lands in `~/`), then invoke it **by full path** ‚Äî a `~/.<tool>` install isn't on a bash call's PATH. Don't retry the same install with sudo.
3. **Run the default login ONCE and keep that one process alive until the human approves.** The login starts a local callback server (e.g. `localhost:4321`); if its process dies ‚Äî **or another app already holds that port** ‚Äî the human's click 404s with the *other app's* error (check it's free first: `lsof -i :<port>`). So run it **foreground with a long timeout**, or as a *persistent* background job (`run_in_background`, `nohup ‚Ä¶ & disown`) ‚Äî for a **localhost-callback** CLI **never** use a bare `‚Ä¶ &` that ends with the shell call (that kills the callback server). Device-code / remote-poll CLIs (Render, Fly, Netlify) have no local server, so a bare `&` is harmless there ‚Äî but `run_in_background` is still better, since it lets you surface the device code before the human approves. Don't pass popup-suppressing flags (`--no-open`, `BROWSER=none`):
   ```bash
   npx -y <pkg>@latest login        # popup opens immediately; call returns on success (timeout ~300s)
   ```
   Backgrounded (`run_in_background`) returns immediately with no inline output ‚Äî poll its output file (bounded loop) for the URL then the success line / process-exit. **Don't add your own `>` redirect**: the harness already captures stdout to the task output file, so a redirect leaves *that* file empty and you'll poll the wrong place and see nothing. Only a *bare top-level* `sleep` is blocked here; `/bin/sleep` or a `sleep` *inside* a loop works (or the Monitor tool).
4. **The popup opens in the human's browser automatically.** Tell them: "approve the login in the window that just opened." **If a CLI prints a URL but doesn't open it itself** (Stripe's `--non-interactive`, üîó print-URL rows), `open '<url>'` (macOS) / `xdg-open '<url>'` (Linux) it **yourself** on a local host so the human still gets the auto-open ‚Äî don't just paste it and make them open it. (Open only the human-facing auth URL ‚Äî never a `secret=`/poll URL; see the Stripe row.) If it didn't open (headless), surface the URL ‚Äî **except for localhost-callback CLIs** (Sanity `:4321`, wrangler `:8976`, ‚Ä¶): on a remote/headless host the human's browser can't reach *this* box's `localhost`, so the pasted URL never completes the callback. There, skip the popup and use the token/env path instead. For device-code flows (Vercel, Stripe, Auth0, Render, WorkOS, Convex) the CLI prints a `user_code` ‚Äî read it from the output and echo it to the human so they can confirm the match before approving. **Spell out the human's exact action ‚Äî never leave them guessing.** Say "approve in the window that just opened," and if a code is shown, state which kind it is: a **visual match** to confirm against the terminal then approve in the browser (device-code flows ‚Äî Vercel/Stripe/WorkOS/Render: nothing to send back), or a **paste-back** where the browser prints a code the human must enter *into the CLI* (Supabase: "Enter this verification code in Supabase CLI") ‚Äî there you ask the human to relay it and you `send` it into the session. Silently polling while the human wonders what to do is a failure. **Tell them to sign into the *right account* in the browser first** ‚Äî a popup approved under the wrong account still "logs in" but stores an `Unauthorized` token. **Send this as you launch the login command ‚Äî don't block waiting for a reply** (most of the time they're already on the right account): localhost-callback / poll flows (wrangler, Netlify) can complete the instant they approve, before a *later* message reaches them, so prime them up front rather than warning after. When you ask for a relayed code, ask for **"the code, or any error text on the page"** ‚Äî they may be staring at an error, and **you can't see screenshots** (if they send an image, say so and ask them to type it).
5. **Wait for the success line ‚Äî or just the process exiting (some CLIs print none) ‚Äî then verify** (whoami/status). Don't blind-`tail` the log ‚Äî the identity line is usually *first* (grep for it); spinner/device-code logs flood with ANSI frames, so trust the verify command's identity *output*, not a log grep. Don't trust the verify **exit code** alone either ‚Äî some (`netlify status`) return an unreliable exit code outside a linked project even when logged in (seen both `0` and non-zero); parse stdout for the identity block. **Verify the *identity*, not just exit-0 ‚Äî and don't say "logged in" until the verify command prints it.** An `Unauthorized`/empty verify *right after* a success line almost always means the human approved in the **wrong account** (or the provider hit a token cap), NOT a stale token ‚Äî ask which; don't dive into a keychain/config diagnostic loop.

**Run the login once and wait ‚Äî don't fire it repeatedly.** It opens a local callback server and blocks until the human approves; spawning extra logins because you didn't see instant success just leaves competing servers behind. One login, keep it alive, wait for the success line.

**Stay until the login command actually returns.** A login ‚Äî especially a *poll-to-complete* step (`stripe login --complete`, `supabase login`) ‚Äî isn't done until the command exits success / prints its identity line. Don't fire it and end your turn (or background it and move on): the human then approves into a dead process and nothing is saved. Block in the foreground, or `run_in_background` **and keep actively polling its output until you see completion**, before you verify.

**Keep ONE session; restarting is the last resort.** Before you stop+restart a login, confirm the session is actually dead (`noninteractive read <sess>`) ‚Äî a restart orphans the human's in-flight approval and burns the code's short TTL. If it's alive (sitting at a code or retry prompt), **reuse it** ‚Äî ask for a fresh code in the tab already open. Only restart when it truly exited or was never created ‚Äî **but** a `Could not create CLI login session` error on a *second* tab while a valid code still shows on the *first* is a `noninteractive` **re-open phantom** (it rescans its whole buffer and re-opens the already-consumed auth URL), NOT a dead session: do NOT restart, use the code tab. When a restart is genuinely needed, **tell the human first** ("that tab's dead, a new one will open"); on success, tell them any other tabs that opened (error or docs pages) are safe to close ‚Äî there are usually extras you can't see. And **never go silent** ‚Äî say "got it, submitting‚Ä¶" when a code arrives, and "still waiting on the browser" if a poll drags, so the human isn't guessing through a 12-call silence. **The moment the login command returns success, fire a quick "got it ‚Äî verifying" *before* you run the verify step and write up your report** ‚Äî the silence while you verify + summarize, right after the human approved, reads as "stuck" to them.

**Logging out for a clean state** (when a task must start logged-out): prefer `<cli> logout`; if it errors `unknown command` (Render has none), delete the config file to force logged-out (`rm ~/.render/cli.yaml`). An already-clean result is SUCCESS, not failure ‚Äî `No login credentials found` (Sanity), `no profiles found` (Daytona). A logout that exits 0 (or prints `Logged out successfully`) is itself proof of clean state ‚Äî go straight to login; don't spend a call on whoami/status/debug to re-confirm logged-out (that verify is for *after* login). **On auto-relogin CLIs (Vercel, Neon) this re-check is worse than wasteful ‚Äî a logged-out `whoami`/`me`/status *itself* launches the OAuth flow and can silently re-log-you-in off a live browser session, so you never reach a clean state and a stray device tab pops. Never chain `logout` and a check in one shell line there** (`‚Ä¶ logout; ‚Ä¶ whoami`) ‚Äî you can't react between them, and the whoami re-auths before you'd see the logout succeeded. Trust the logout's own success line.

**If the browser login fails, read the CLI's error and follow its hint ‚Äî don't blindly retry.** Many CLIs offer a device-code mode (e.g. `railway login --browserless`): it prints a URL + pairing code instead of using a localhost callback (more robust, works headless). Device-code mode usually needs a real TTY, so drive it with `noninteractive` ‚Äî for the railway example that's `npx noninteractive start railway login --browserless` (railway's a *binary* CLI, so the `start` form runs the literal command; the bare `npx noninteractive <pkg> ‚Ä¶` form is for npm CLIs and would here run `npx railway`, fetching the wrong package ‚Äî see the two-form rule below) ‚Äî and read the URL/code from the returned `urls` array. Token env vars are the last-resort no-popup path.

**Some logins gate the popup behind an interactive prompt** ("Press any key‚Ä¶", an auth-method picker, a device-name field) and *fatal* when backgrounded ‚Äî Railway, Daytona, Auth0, Supabase, Convex. Drive the gate in a PTY, two ways:

- **`noninteractive`** (also auto-opens the OAuth URL it finds): `npx noninteractive <pkg> login` for an npm CLI, or `npx noninteractive start <binary> login` for a binary/brew CLI (`start` runs the literal command, not `npx`) ‚Äî **when `<binary>` is a full path, add `--name <short>`** (`start --name auth0 /path/to/auth0 login`): the session name auto-derived from a path mangles into a garbage name that fails to start. **Then send the keystroke** ‚Äî `npx noninteractive send <sess> ' '` (or `''` for Enter) ‚Äî and `read --wait` for the URL/success. Just starting it leaves the login parked at the gate (the #1 failure). The `send`'s own immediate echo can still show the **pre-keystroke** buffer (e.g. the TUI picker unchanged) ‚Äî don't read "keystroke failed" into that and re-send (you'll over-advance a menu); `read --wait` once more and the advance/URL appears. Flags (`--timeout`/`--no-open`) go on the `send`/`read` calls, NOT trailing the login command (they fatal `unknown flag`). It auto-opens **every** URL it prints ‚Äî the auth popup is the first `‚Ä¶authorize‚Ä¶`; a later release/docs link is a junk tab. **If the CLI *actually* opens its own browser under the PTY** (Railway v5 does ‚Äî confirmed because `expect` alone, which never auto-opens, still yields one tab), noninteractive's auto-open is **redundant ‚Üí a duplicate tab**; `--no-open` lets the CLI's own open stand, or use `expect`. **But verify it truly self-opens first** ‚Äî some CLIs only *print* `opened browser window` while noninteractive does the real opening (Auth0: under the PTY `--no-open` yielded **0** tabs ‚Äî it kills the device page and breaks the login). When unsure, leave auto-open on and just tell the human to ignore the junk tabs. It also rescans its **whole** buffer on each `send`, so it **re-opens** URLs ‚Äî a re-opened, already-consumed auth URL renders a *phantom* `Could not create CLI login session` tab even though your login is fine. **You can't self-count windows**: to report what opened, count the `[opened: <url>]` lines in the tool results, and expect ‚â•1 junk/phantom tab on these flows. **Completion** is stamped `[exited <N>]` + `[session complete]` in the buffer right after the CLI's own success line (which is often NOT the word "logged in" ‚Äî e.g. Convex prints `Saved credentials`); match those markers / the CLI's real success string, never a generic `logged in` regex. Device-code CLIs (Render) need no keystroke (the popup opens on `start`).
- **`expect`** bundles spawn + keystroke + success/error exit in one line:

```bash
expect -c 'spawn <cli> login; set timeout 180; expect {
  -nocase -re "(press enter|any key|open the browser|open browser|\(y/n\)|Login with Browser)" {send "\r"; exp_continue}
  -nocase -re "(error|denied|fail|unsuccessful|not logged in|not authenticated|no token|no code)" {exit 1}
  -nocase -re "(logged in|success|authenticated)" {exit 0}
  timeout {exit 2}
  eof {catch wait r; exit [lindex $r 3]} }'
```
On `eof`, propagate the child's **real exit code** (`catch wait r; exit [lindex $r 3]`), not a blanket `exit 0` ‚Äî `eof` only means the child closed its fd, so a bare `exit 0` reports success even when the CLI died non-zero with an error your narrow regex missed (`unauthorized`, `connection refused`). A clean exit (0) still passes; a non-zero exit now correctly fails. Verify with whoami/status regardless. **The failure/negation pattern is listed BEFORE the success one on purpose** ‚Äî `expect` evaluates top-to-bottom, and the success strings are unanchored substrings (`logged in` sits inside `not logged in`, `success` inside `unsuccessful`), so a negated/failed line must hit `exit 1` first or it'd false-`exit 0`. **The success alternation is also kept narrow and non-exhaustive** ‚Äî CLIs print wildly varied success strings (Railway `Signed in as`, Convex `Saved credentials`), but don't widen it to chase every variant (each one you add is another negation-substring risk): `eof`‚Üíexit-code is the load-bearing signal (it caught Railway's `Signed in as` fine in testing), and the success regex only lets a CLI that *lingers* after success exit early. **Auth-method / TUI pickers are NOT in the match list** ‚Äî Auth0's *first* prompt is a `How would you like to authenticate?` survey (`As a user` / `As a machine`), and Daytona has a TUI picker; the generic regex above never fires on them, so expect parks at the picker until `timeout` and the browser never opens. Drive those with the **`noninteractive` path** (Auth0's documented recipe sends two Enters ‚Äî one to pick `As a user`, one to open the browser ‚Äî see its row; Daytona's is a redraw-heavy **TUI** picker that an `expect` text-regex won't reliably match at all, so noninteractive is the answer there, not a regex tweak). Only if you must use expect on a *plain-text* survey like Auth0's, add **that CLI's own picker prompt** to the press-enter alternation (Auth0: `how would you like to authenticate`) so it sends `\r` and accepts the highlighted default (`As a user`) ‚Äî match each CLI's actual prompt string, and don't try this on a full TUI picker. Prefer inline `expect -c '‚Ä¶'` over a `.exp` file (Write won't clobber an unread scratchpad file).

**A few CLIs auto-switch to a JSON/agent mode under non-TTY instead of gating** ‚Äî don't wrap these in `expect`. Stripe v1.40.9+ is the case: `stripe login --non-interactive` emits a `browser_url`+`verification_code`+`next_step` JSON and **exits 0 ‚Äî that's step 1, not success**; surface the code, run the `next_step` (`stripe login --complete '<url>'`), then verify.

**OS note ‚Äî validated on macOS.** The login commands, the üü¢/üîµ/‚å®Ô∏è/üîó/üîë classes, and the `expect`/`noninteractive` recipes are OS-agnostic. What differs on **Linux**: **install** (use each provider's docs-URL method / system package manager / `curl` installer ‚Äî not `brew` or Darwin-arm64 tarballs) and **paths** ‚Äî user binaries live under `~/.local/bin`, config under `~/.config` (XDG), not `/opt/homebrew/bin` or `~/Library/‚Ä¶`. Invoke a CLI by its name on PATH (or `command -v <cli>`); don't hard-code `/opt/homebrew`.

**Per-provider commands for 19 CLIs** (exact login command, TTY caveats, verify, token env): see [`references/providers.md`](references/providers.md). Sanity and Netlify are below.

## Sanity  (popup: auto-opens ¬∑ login: automatic ¬∑ verified)

- Install: `npx -y @sanity/cli@latest` (bin `sanity`; `@sanity/cli` is lighter than the `sanity` meta-pkg and has full `login`). In a Studio project, `npx sanity login` is instant.
- Login: `npx -y @sanity/cli@latest login --provider google` (swap `google`‚Üî`github` for the human's account). **Pass `--provider`** ‚Äî bare `sanity login` shows an interactive picker that hangs when backgrounded. Run it once, persistent; the popup opens (`Opening browser at <url>`) and the `localhost:4321` callback must stay reachable until approval.
- Success: exits 0 (callback server shuts down) and prints `Login successful` on current versions ‚Äî but detect completion by the exit + `sanity debug` (authoritative), not by grepping the string. `logout` prints `Logged out successfully`; `No login credentials found` only shows when *already* logged out (both are clean states).
- Verify: `npx -y @sanity/cli@latest debug` ‚Üí a multiline `User:` block; grep **`Email:`** for the address (the `User:` line is just a section header and has no email on it). (`sanity whoami` doesn't exist.)
- CI / no popup: `export SANITY_AUTH_TOKEN=<token>` (read before config), or `sanity login --with-token < token.txt`.

## Netlify  (popup: auto-opens ¬∑ login: automatic)

- Install: `npx -y netlify-cli` (NOT `npx netlify` ‚Äî that's the JS API client). Bin `netlify`/`ntl`. `NETLIFY_TELEMETRY_DISABLED=1` skips the telemetry prompt.
- Login: `npx -y netlify-cli login`. Default opens the popup and polls Netlify's API (no localhost callback to collide), stores the token, prints `You are now logged into your Netlify account!`. If a token is already cached it short-circuits `Already logged in ‚Ä¶`. Do NOT use `--request` (the copy-a-URL ticket flow) or `BROWSER=none`.
- Verify: `npx -y netlify-cli status` ‚Üí "Current Netlify User" block. Its **exit code outside a linked project is unreliable** ‚Äî seen both `0` and `1` even when logged in (alongside an `Error: ‚Ä¶ not linked to a project` line) ‚Äî so parse stdout for the block, don't trust the code. (`--json` avoids the ANSI color codes in the plain output.)
- CI / no popup: `export NETLIFY_AUTH_TOKEN=<pat>` or `--auth <token>` on any command (PAT from app.netlify.com/user/applications).
